# Firebase Schema & Rules

The game uses a **hybrid** Firebase backend. Rules for both databases live in version control and
are deployed by CI (`firebase deploy --only hosting,firestore,database`). See
`.agent/workflows/deploy.md`.

## Firestore — session metadata & game state
Collection `sessions/{code}` where `{code}` is a 6-digit string. One document per active session:

| Field          | Type      | Written by | Notes                                            |
|----------------|-----------|------------|--------------------------------------------------|
| `created`      | timestamp | desktop    | server timestamp at creation                     |
| `connected`    | boolean   | both       | desktop sets false on create; mobile sets true   |
| `gameState`    | map       | desktop    | `{ active, score, state }`                        |
| `gameAction`   | string    | mobile     | `start` / `restart`; desktop clears it to `null` |
| `lastActivity` | timestamp | both       | server timestamp; touched on each write          |
| `version`      | number    | desktop    | debug marker                                      |
| `feedback`     | map       | desktop    | `{ type, at }` one-shot haptic cue for the phone  |

Rules (`firestore.rules`): reads/deletes require a 6-digit `{code}`; creates/updates additionally
require the document to contain only the keys above. All other collections are denied. There is no
auth yet, so access is path/shape-scoped, not per-user. If Anonymous Auth is added, store an
`ownerId` on create and require `request.auth.uid == ownerId`.

### Multiplayer (v2) session fields
Multiplayer sessions carry `mode: 'multi'` plus the fields below — all **optional** in the rules,
so the schema is a strict superset of the legacy solo doc (rollout-safe in any deploy order). Slots
are validated for **p1–p6** even though the app caps at `gameConfig.maxPlayers` (currently 3), so
raising the cap needs no rules change.

| Field         | Type   | Written by | Notes                                                                  |
|---------------|--------|------------|------------------------------------------------------------------------|
| `mode`        | string | desktop    | literally `'multi'`; absence = legacy solo doc                          |
| `players`     | map    | both       | `{p1..p6: {name 1–16, token ≤40, score 0–100000, alive, connected, joinedAt, death?}}`; phone writes its own entry at claim (transaction), desktop owns score/alive/death |
| `gameActions` | map    | both       | `{p1..p6: null \| 'start' \| 'restart'}`; phone sets its slot, desktop clears |
| `feedback`    | map    | desktop    | per-slot `{type, at}` one-shot haptics (only the targeted phone reacts) |
| `results`     | map    | desktop    | `{winnerSlot, endedAt, defeated:[{slot,name,score,death}]}`; null while playing |

`results` and per-player `death` are **shallowly** validated (desktop-written display data — deep
array validation in rules is brittle; accepted risk).

## Realtime Database — joystick stream
Path `controllers/{code}`. High-frequency input, kept out of Firestore to avoid write costs.

**Legacy solo layout** (flat, written by the single phone):

| Field         | Type    | Notes                                            |
|---------------|---------|--------------------------------------------------|
| `connected`   | boolean |                                                  |
| `joystick`    | map     | `{ x, y }`, each a number in roughly [-1, 1]     |
| `timestamp`   | number  | `ServerValue.TIMESTAMP`                          |
| `initialized` | boolean | set by desktop on create                         |

**Multiplayer layout** (per-slot children, one per phone): `controllers/{code}/{slot}` where
`{slot}` matches `^p[1-6]$`, each holding `{connected, joystick{x,y}, timestamp}`. Each phone
registers `onDisconnect().remove()` on ITS slot child, so a vanished phone frees only its own
input; the desktop watches the parent and reconciles the roster.

Rules (`database.rules.json`): read/write require a 6-digit `{code}`; the legacy flat keys keep
their child `.validate` rules, and a single `$slot` wildcard validates the per-slot children
(key must match `p[1-6]`; same joystick ranges; unknown keys rejected). Both layouts validate, so
old cached clients and new clients coexist during rollout.

## Firestore — global leaderboard
Collection `leaderboard/{playerId}` where `{playerId}` is a stable per-device UUID generated by
the client (`localStorage` key `snake_player_id`). One row per device:

| Field       | Type      | Notes                                                       |
|-------------|-----------|-------------------------------------------------------------|
| `name`      | string    | the player's handle, 1–16 chars (sanitized client + rules)  |
| `score`     | number    | the device's best score, 0–100000, monotonic (never lowers) |
| `updatedAt` | timestamp | must equal `request.time` on every write                    |

Rules (`firestore.rules`): public reads (anyone can view the board); creates/updates require
exactly the keys above with the validations noted; deletes are denied. Reads use the single-field
`score` index (`orderBy('score','desc')` for the Top-N, `where('score','>')` capped at 101 docs
for rank) — no composite index needed.

## Cleanup / lifecycle
Sessions are ephemeral. The desktop host registers `onDisconnect().remove()` on its RTDB node and
best-effort-deletes the Firestore doc + RTDB node on `beforeunload`. For guaranteed Firestore
cleanup, configure a **TTL policy** on the `lastActivity` field (Firebase console → Firestore →
TTL) so abandoned sessions expire automatically.

## Security model & accepted risks
There is **no authentication** — access is scoped by path shape (6-digit code) and document
shape, not identity. The following risks are **known and accepted** for this project's scale:

- **Session griefing:** anyone who guesses/enumerates an active 6-digit code can read the
  session, inject joystick input, send start/restart actions, or delete the session doc. The
  open `delete` grant cannot be removed — the host's own `beforeunload` cleanup depends on it.
  Impact is annoyance only; no user data is at stake.
- **Leaderboard spoofing:** writes are shape/range-validated and monotonic, but any client can
  submit an in-range fake score or create rows under arbitrary device ids.
- **Multiplayer slot griefing:** the per-slot rejoin `token` in `players.{slot}` is readable by
  anyone holding the code; a malicious client can overwrite roster entries, fake actions, or spoof
  another slot's joystick. Same griefing class as session griefing — annoyance only.
- **Stale-client RTDB clobber:** an old cached phone joining a multiplayer session writes the
  legacy flat shape over `controllers/{code}`, transiently wiping the slot children; each live
  phone's next ~33ms joystick update recreates its child (self-heals). The SW cache bump shipped
  with the feature shrinks this skew window to one reload.

**Mitigations (console-side, owner action):** Firestore TTL on `sessions.lastActivity`, a
billing **budget alert** as the cost tripwire, and — if real traffic ever arrives — Firebase
**App Check** plus Anonymous Auth with an `ownerId` on session create.
